## Issue 1 — `/beton-cire-ral-kleuren/` returns 404

### What's happening
The URL `https://beton-cire-webshop.nl/beton-cire-ral-kleuren/` returns **HTTP 404**. The page it was meant to reach still exists, but its slug was changed to `ral-kleuren`, so the canonical URL is now `https://beton-cire-webshop.nl/ral-kleuren/` (HTTP 200).

No 301 redirect was configured when the slug was changed, which means:
- Visitors clicking old internal or external links land on a 404.
- Googlebot sees a dead link instead of a permanent move → the link equity is lost.

### Where the broken URL still appears (production DB scan)

| Type | ID | Title | URL |
| --- | --- | --- | --- |
| Page | 17230 | Beton Cire Kleuren | https://beton-cire-webshop.nl/beton-cire-kleuren/ |
| Page | 18304 | Kleurstalen Betonlook Verf Aanvragen! | https://beton-cire-webshop.nl/kleurstalen-betonlook/ |
| Page | 14335 | Original kleurstalen aanvragen | https://beton-cire-webshop.nl/original-kleurstalen/ |
| Page | 14343 | Velvet kleurstalen aanvragen | https://beton-cire-webshop.nl/velvet-kleurstalen/ |
| UX template | 19729 | Kleuren | — (embedded block) |
| UX template | 14351 | MV1Text | — (embedded block) |

### Recommended fix
1. **Add a permanent 301 redirect** from `/beton-cire-ral-kleuren/` to `/ral-kleuren/`. Easiest implementation: WordPress plugin *Redirection* (Tools → Redirection), or add this to `.htaccess`:
   ```apache
   Redirect 301 /beton-cire-ral-kleuren/ /ral-kleuren/
   ```
2. **Update the internal links** on the 4 pages + 2 ux_templates listed above so they point directly to `/ral-kleuren/`. A direct link is better than relying on a redirect.
3. **Re-submit the URL in Google Search Console** → *URL Inspection* → *Request Indexing* for the canonical `/ral-kleuren/`.

## Issue 2 — `/beton-cire-easyline-kleuren/` returns 301 (redirect chain)

### What's happening
`https://beton-cire-webshop.nl/beton-cire-easyline-kleuren/` returns **HTTP 301** with `Location: https://beton-cire-webshop.nl/product-categorie/beton-cire/easyline/`.

The redirect itself is fine (not a 404), but internal links should always point **directly at the final destination**. Every internal link that goes through a redirect:
- costs one extra request round-trip (slower page load);
- slightly dilutes link equity;
- increases crawl budget without benefit.

### Recommended fix
Update any internal links that reference `/beton-cire-easyline-kleuren/` to use the direct URL `/product-categorie/beton-cire/easyline/` instead. Keep the 301 in place for external inbound links and saved bookmarks.

## Scan methodology

- Fetched each audited page HTML, extracted every `href` inside the `<main>` element.
- For each unique internal URL, issued a `HEAD` request and recorded the status code.
- Followed 301s one hop to verify the final destination is 200.
- Queried the WordPress posts table (`post_content LIKE '%beton-cire-ral-kleuren%'`) to find every location the broken URL still lives on production.

## Issue — `/nieuwsbrief/` has no landing page (2026-04-19)

### What's happening
While rebuilding the footer, the live site copy listed a "Nieuwsbrief" (newsletter) link. There is currently **no published page** with slug `nieuwsbrief`, `newsletter`, `aanmelden`, or `nieuwsbrieven` on staging or production — so the link would 404.

### Decision
Link removed from the footer until a dedicated subscribe page is created. No redirect added because there is no historical destination to 301 from — this is a new concept, not a moved page.

### Follow-up
When the marketing team wants to activate the newsletter signup, create a page (or dedicated block on an existing page) and re-add the footer entry pointing to it.
